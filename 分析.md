# TreeSearch 核心模块与核心算法分析

> 说明：本文主要分析 `treesearch/` 这条 Python 检索主线的核心模块与算法实现，不覆盖离线 Go 排障运行时。
>
> 如果你要看这次新增的 Go 侧常驻排障能力，请优先参考：
> - `examples/troubleshooting/README.md`
> - `examples/troubleshooting/AGENTS.md`

本文基于本仓库源码，对 TreeSearch（`pytreesearch`）的核心模块职责与核心算法实现做结构化梳理，并附上关键代码入口，方便按图索骥。

> 目标一句话：把文档/代码解析成“树节点”（node）作为最小检索单元写入 SQLite（含 FTS5），查询时做“文档路由 + 节点预评分（FTS5/BM25 + Grep）+ 合并输出”。

## 1. 核心模块分工（按执行路径）

- 对外统一 API：`treesearch/treesearch.py`
  - `TreeSearch`：懒加载索引、增量重建、调用统一 `search()`。
- 索引构建（parse -> tree）：`treesearch/indexer.py`
  - `build_index()` 批量建索引；`md_to_tree()` / `text_to_tree()` / `code_to_tree()` / `json_to_tree()` / `csv_to_tree()` 生成树结构；`_finalize_tree()` 统一后处理。
- 解析器注册与路由：`treesearch/parsers/registry.py`
  - `ParserRegistry`、`SOURCE_TYPE_MAP`（ext -> source_type）、`PREFILTER_ROUTING`（source_type -> 推荐 prefilter 链）。
- 数据模型与树工具：`treesearch/tree.py`
  - `Document`（包含 node_id -> node dict 的 O(1) 缓存 map）、`flatten_tree()`、`build_tree_maps()`、`load_documents()`。
- 存储/倒排索引/打分引擎：`treesearch/fts.py`
  - `FTS5Index`：建表、写入 nodes/fts_nodes/documents、FTS5 BM25 排名、LIKE fallback、`score_nodes()` 的归一化与祖先传播。
- 统一搜索管线：`treesearch/search.py`
  - `search()`、`GrepFilter`（精确匹配）、`_CombinedScorer`（分数求和）、`_merge_doc_results()`（多文档合并）、`_attach_node_fields()`（回填 text/summary/行号/祖先标题）。
- 配置与分词：`treesearch/config.py`、`treesearch/tokenizer.py`
  - `TreeSearchConfig` 默认参数；CJK tokenization（`auto/jieba/bigram/char`）。
- CLI：`treesearch/cli.py`
  - `treesearch index/search` 子命令对接 `build_index()` / `load_documents()` / `search()`。

## 2. 核心算法 1：文件 -> 树（Indexing / Tree Building）

TreeSearch 的“结构化”主要来自索引阶段：把源文件切分成多级标题/结构节点，再建成树。

### 2.1 Markdown：标题切分 + 栈建树

- 入口：`treesearch/indexer.py:md_to_tree()`
- 步骤：
  1. `_extract_md_headings(content)`：用正则识别 `#{1,6}` 标题；在 code fence（```）内不识别。
  2. `_cut_md_text(markers, lines)`：按相邻标题的行区间切出每个节点的 `text`，并带上 `level/line_start/line_end`。
  3. 可选瘦身：`_update_token_counts()` + `_thin_tree()`（见 2.4）。
  4. `_build_tree(nodes)`：栈算法把扁平节点按 `level` 组织成父子结构（见 2.3）。
  5. `_finalize_tree()`：分配 node_id、格式化字段、生成 summary/description。

### 2.2 纯文本：规则标题识别 + 栈建树

- 入口：`treesearch/indexer.py:text_to_tree()`
- 标题识别：`_detect_headings(lines)` 用纯规则匹配：
  - 数字层级（例如 `1.2.3 Title`）、罗马数字（`IV.`）、字母（`A)`）、RST 下划线样式、全大写短行等
  - 中文章节样式（`第X章/节/篇/部`、`一、`、`（1）` 等）
- 若完全检测不到标题：回退为“单根节点”。

### 2.3 扁平节点 -> 树：栈算法（核心结构构建）

- 实现：`treesearch/indexer.py:_build_tree(node_list)`
- 机制：
  - 维护 `stack[(tree_node, level)]`。
  - 每来一个节点：弹出所有 `stack[-1].level >= current_level`；
    - 栈空：成为 root
    - 否则：挂到栈顶节点的 `nodes` 子列表
  - 将当前节点压栈。

注意：`_build_tree()` 里会生成一个临时 `node_id` 计数器，但最终会被 `_finalize_tree()` 里的 `assign_node_ids()` 重新分配（因此“最终 node_id 是连续编号”）。

### 2.4 Thinning（合并过碎的小节点）

- 实现：`treesearch/indexer.py:_thin_tree(node_list, min_tokens)`
- 目的：减少“非常小的子树节点”，把它们内容合并进父节点，降低碎片化带来的 BM25 偏置/噪声。
- 步骤：
  1. `_update_token_counts()`：从后往前累加“自己 + 子孙”的文本，算 `text_token_count`。
  2. `_thin_tree()`：若某节点 `text_token_count < min_tokens`，就把其子节点 `text` 追加合并到父节点，并移除这些子节点。

### 2.5 Code：AST/正则抽取结构

- 入口：`treesearch/indexer.py:code_to_tree()`
- 结构抽取：`_detect_code_headings(lines, ext, source)`
  - `.py`：优先 `treesearch/parsers/ast_parser.py:parse_python_structure(source)`（提取 class/def 结构与签名）；失败则正则兜底。
  - 其他语言：按扩展名使用不同的正则模式抓取 class/interface/method/func。
- 后续同样走“切片 + thinning + 栈建树 + finalize”。

## 3. 核心算法 2：树节点 -> SQLite FTS5（持久化与倒排索引）

### 3.1 DB schema（核心表）

`treesearch/fts.py:FTS5Index._init_db()` 创建并维护所有表：

- `documents`：文档元信息 + 整棵树持久化
  - 关键列：`structure_json`（整棵树 JSON）、`doc_description`、`source_path`、`source_type`、`index_hash`
- `nodes`：节点元信息（结构字段）
  - 关键列：`doc_id, node_id, title, summary, depth, line_start, line_end, parent_node_id, content_hash`
- `fts_nodes`：可检索文本载体
  - FTS5 可用时：FTS5 虚表（`tokenize='unicode61 remove_diacritics 2'`）
  - 不可用时：普通表，搜索走 LIKE fallback
  - 关键列：`title, summary, body, code_blocks, front_matter`
- `index_meta`：增量索引的 file hash
  - `source_path -> file_hash(mtime_ns:size)`

### 3.2 Indexing：如何把 tree 写入 FTS

- 入口：`treesearch/fts.py:FTS5Index.index_document(document, force=False)`
- 关键点：
  - 增量跳过：对 `documents.index_hash`（整棵结构 JSON 的 md5）做短路；若 hash 未变且 `force=False`，就跳过。
  - 结构字段入库：用 `treesearch/tree.py:build_tree_maps(structure)` 计算 `parent_map/depth_map` 写入 `nodes`。
  - 文本字段入 FTS：对每个 node 的 `text` 调 `parse_md_node_text()` 拆成：
    - `front_matter`（YAML front matter）
    - `body`
    - `code_blocks`
    再连同 `title/summary` 一起写入 `fts_nodes` 的不同列，以便列级加权。

### 3.3 Tokenization：CJK 预分词 + 英文依赖 unicode61

- 实现：`treesearch/fts.py:_tokenize_for_fts(text)`
  - 若检测到 CJK 字符：调用 `treesearch/tokenizer.py:tokenize()` 生成 tokens，并用空格拼接（使 FTS5 能按词命中）。
  - 若非 CJK：原样写入，依赖 FTS5 的 `unicode61` tokenizer。

## 4. 核心算法 3：查询管线（Routing -> Prefilter -> Scoring -> Merge）

统一入口：`treesearch/search.py:search(query, documents, ...)`

### 4.1 Stage 1：文档路由（多文档时）

- `FTS5Index.search_with_aggregation(query, top_k=top_k_docs)`
  - 先抓节点命中，再按 doc 聚合 `hit_count/best_score/avg_score`。
  - 以 `best_score` 排序取 Top-K 文档，实现“先找相关文档，再做细粒度节点检索”。

### 4.2 Stage 1.5：Prefilter 链路选择（Grep/FTS 组合）

- 路由规则：`treesearch/parsers/registry.py:PREFILTER_ROUTING`
  - `code` 默认 `['grep', 'fts5']`
  - `markdown/text/...` 多为 `['fts5']`
- 实现方式（非常关键）：
  - `treesearch/search.py:GrepFilter` 是“精确匹配打分器”，不是硬过滤器。
  - `treesearch/search.py:_CombinedScorer` 会把多个 scorer 的分数做“逐 node 相加”。

### 4.3 Stage 2：节点打分（node-level）

FTS5/BM25：`treesearch/fts.py:FTS5Index.search()`

- `MATCH` 表达式：
  - 无 `fts_expression` 时：对 query 分词清洗，多词默认 `OR` 提升 recall。
  - 有 `fts_expression` 时：`_tokenize_fts_expression()` 保留 `AND/OR/NOT/NEAR`，仅对 term 分词。
- 排名：SQL 用 `bm25(fts_nodes, w_title, w_summary, w_body, w_code, w_front_matter)`
  - 权重定义：`treesearch/fts.py:_DEFAULT_WEIGHTS`（title/body/summary/code/front_matter）
  - 分数方向：bm25() 更小更相关，代码取反：`fts_score = -rank_score`。
- phrase boost：对多词 query 尝试 quoted phrase 匹配，命中的 node 乘以 1.5，再重排。

FTS 作为 PreFilter：`treesearch/fts.py:FTS5Index.score_nodes()`

- 将 top200 的 `fts_score` 归一化到 `[0,1]`。
- 祖先传播：用 `nodes.parent_node_id` 做一次 bottom-up：父节点加上 `ancestor_decay * max(child_scores)`（默认 0.6），再归一化。

Grep：`treesearch/search.py:GrepFilter.score_nodes()`

- 遍历树；若 `title/summary/text` 包含 query（或 regex 匹配），该 `node_id` 得 1.0。

### 4.4 Stage 3：结果回填与合并

- `_attach_node_fields()`：按 `text_mode(full/summary/none)` 回填 text；可选 `include_ancestors` 时把祖先标题链附加。
- `_merge_doc_results(merge_strategy)`：
  - `per_doc`：保留按 doc 分组
  - `global_score`：节点全局排序后再按 doc 重新分组
  - `interleave`（默认）：主要是过滤空 doc（并不做真正节点级交错）

## 5. 增量索引与懒加载（用户体验层的核心）

- 批量索引增量跳过：`treesearch/indexer.py:build_index()`
  - `_file_hash(fp) = mtime_ns:size`
  - `FTS5Index.get_all_index_meta()` 一次性取回所有 hash，用于批量判定。
  - 只对变更文件重建，再 `set_index_meta_batch()` 一次性写回。

- `TreeSearch.asearch()` 懒加载：`treesearch/treesearch.py`
  - 如果存在 `db_path` 且无变更：直接 `FTS5Index.load_all_documents()`/`load_documents()` 复用。
  - 如有变更：只对变更文件 `aindex()`，然后 reload。

## 6. 重要实现细节与容易误解点

- TreeSearch 返回的“检索实体”是节点（node），不是整篇文档；多文档时先路由再返回每 doc 的 top nodes。
- “结构感”的关键不在搜索时做树遍历，而在：
  - 索引时按标题/结构切分成节点
  - `FTS5Index.score_nodes()` 做祖先传播，让深层命中能带出父标题上下文
- 代码检索默认叠加 Grep 与 FTS（分数求和），因此 grep 命中会显著影响排序。
- `TreeSearchConfig` 里有 `fts_*_weight/fts_enabled/fts_auto_index` 字段，但当前源码没有把这些配置真正传入 `FTS5Index(weights=...)`（权重仍用 `fts.py:_DEFAULT_WEIGHTS`）。

## 7. 关键入口速查

- 索引入口：`treesearch/indexer.py:build_index()`
- Tree 构建：`treesearch/indexer.py:_build_tree()`
- Summary/Description：`treesearch/indexer.py:generate_summaries()` / `generate_doc_description()`
- DB schema + FTS：`treesearch/fts.py:FTS5Index._init_db()` / `index_document()` / `search()` / `score_nodes()`
- 搜索管线：`treesearch/search.py:search()` / `GrepFilter` / `_CombinedScorer` / `_merge_doc_results()`
- 解析/预过滤路由：`treesearch/parsers/registry.py:PREFILTER_ROUTING` / `SOURCE_TYPE_MAP`
